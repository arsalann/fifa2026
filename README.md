# World Cup 2026 App

Static React/Vite single-page app for the 2026 FIFA World Cup. It shows groups,
schedule, teams, team detail pages, bracket, Golden Boot standings, and
head-to-head comparison. The browser app is static and has no login, database
credentials, or direct provider API ingestion.

The runtime data contract is app-ready JSON generated by Bruin from MotherDuck:

- `src/data/bruin/schedule.json` - bundled schedule, scores, live status,
  goals, ESPN event IDs, and optional betting insights.
- `src/data/bruin/teams.json` - bundled team metadata, flags, history, titles,
  top scorers, and slugs.
- `src/data/bruin/details.json` - bundled ESPN match summaries, box scores,
  venue details, rosters, and starting lineups.
- `public/data/bruin/*.json` - deploy-time public copies of the same exports,
  used as a refreshable browser fallback.

The original reference files in `src/data/` and
`pipelines/worldcup_2026/reference/` are pipeline inputs. Browser code imports
and fetches only the generated Bruin JSON or the Netlify live endpoint.

## Local App Commands

```bash
npm ci
npm run dev
npm test
npm run build
```

Optional analytics are controlled by Vite environment variables:

- `VITE_POSTHOG_KEY` enables PostHog.
- `VITE_POSTHOG_HOST` overrides the PostHog host.

When `VITE_POSTHOG_KEY` is unset, analytics functions are no-ops and
`posthog-js` is not loaded.

## Replicate and Run This Project

There are three useful replication levels: run the static app from committed
exports, refresh the data pipeline into your own MotherDuck database, and deploy
the full live app on Netlify.

### 1. Run the Static App Locally

This path needs only Node and npm. It uses the committed Bruin JSON files, so no
MotherDuck, Bruin Cloud, ESPN, Kalshi, Polymarket, or Netlify credentials are
required.

Prerequisites:

- Node `20.19.0` or newer.
- npm.

Steps:

```bash
git clone <your-fork-or-repo-url>
cd fifa2026
npm ci
npm run dev
```

Open the Vite URL printed by the command, usually
`http://localhost:5173/`. The app should render from the bundled files in
`src/data/bruin/`.

Before changing or deploying code, run:

```bash
npm test
npm run build
```

### 2. Refresh Data Into Your Own MotherDuck Database

Use this path when you want to reproduce the Bruin-generated JSON yourself.

Prerequisites:

- Bruin CLI installed and available as `bruin`.
- A MotherDuck account and a MotherDuck token.
- Node dependencies installed with `npm ci`.

Steps:

```bash
cp .bruin.yml.example .bruin.yml
export MOTHERDUCK_TOKEN=<your-motherduck-token>

npm run pipeline:validate
npm run pipeline:refresh
```

`pipeline:refresh` runs both Bruin pipelines, writes warehouse tables into the
`fifa` MotherDuck database, then exports JSON into both app data locations:

- `src/data/bruin/`
- `public/data/bruin/`

The generated JSON files are the files the app actually reads. After refreshing
them, run:

```bash
npm test
npm run build
npm run dev
```

Keep `.bruin.yml` and any shell files containing `MOTHERDUCK_TOKEN` out of git.
They contain private credentials.

### 3. Deploy Your Own Netlify Copy

Use this path when you want a hosted copy with the live MotherDuck read
function.

Prerequisites:

- A Netlify site connected to your repository.
- A MotherDuck token available to Netlify as `MOTHERDUCK_TOKEN`.
- Optional Bruin Cloud API credentials if you want the Netlify function to
  trigger the live Bruin pipeline on demand.

Netlify reads `netlify.toml`, so set these build settings:

- Build command: `npm run build`.
- Publish directory: `dist`.
- Node version: `20.19.0`.

Set these Netlify environment variables as needed:

- `MOTHERDUCK_TOKEN` for `/.netlify/functions/bruin-schedule`.
- `MOTHERDUCK_DATABASE` if your database is not named `fifa`.
- `BRUIN_CLOUD_API_KEY` if the function should trigger Bruin Cloud.
- `BRUIN_CLOUD_PROJECT_ID`, `BRUIN_CLOUD_PIPELINE`, and
  `BRUIN_CLOUD_BASE_URL` if your Bruin Cloud project differs from the defaults.
- `VITE_POSTHOG_KEY` and `VITE_POSTHOG_HOST` if you want analytics.

Recommended deploy flow:

```bash
export MOTHERDUCK_TOKEN=<your-motherduck-token>
npm run pipeline:refresh
npm test
npm run build
git add src/data/bruin public/data/bruin README.md
git commit -m "Refresh World Cup app data"
git push
```

Netlify then builds the static app. At runtime, the browser tries the Netlify
function first, then falls back to `public/data/bruin/schedule.json`, then to
the bundled `src/data/bruin/schedule.json` compiled into the app.

### 4. Forking Notes

If you fork or copy the project:

- Update the remote URLs inside the reference SQL assets only if you want them
  to read reference JSON from your fork instead of the upstream repository.
- Keep the generated JSON schema stable unless you also update
  `src/lib/data.jsx`, the pages, and tests.
- Keep all browser-exposed configuration prefixed with `VITE_`; never expose
  `MOTHERDUCK_TOKEN`, `BRUIN_CLOUD_API_KEY`, or `.bruin.yml` to client code.
- Run `npm test`, `npm run build`, and `npm run pipeline:validate` before
  publishing pipeline or runtime changes.

## How the App Works

`src/main.jsx` mounts a `HashRouter` app, so deployed routes work from static
hosting without server-side routing. `src/App.jsx` defines the routes:

- `/` - group tables, today's matches, third-place race, favorite team card,
  and market-derived advancement/continent widgets.
- `/schedule` - weekly or all-match schedule, localized to the user's time
  zone.
- `/teams` - team search/list.
- `/team/:slug` - team profile, group record, fixtures, results, and betting
  markets when available.
- `/bracket` - knockout bracket.
- `/scorers` - Golden Boot table computed from generated match goal data.
- `/compare` - two-team comparison using team metadata and 2026 match state.

`src/lib/data.jsx` owns the schedule data lifecycle:

1. Start immediately with the bundled import from
   `src/data/bruin/schedule.json`, which makes the static build render offline.
2. Try `/.netlify/functions/bruin-schedule` with `cache: no-store`.
3. If the function is unavailable, fetch `public/data/bruin/schedule.json`.
4. If that also fails or returns unusable group data, keep the bundled JSON.
5. Poll every 25 seconds while a match is live or near kickoff, otherwise every
   5 minutes. Focus, online, visibility, pointer, and keyboard activity can also
   trigger a throttled refresh.

The live payload contains only schedule rows. Team metadata and match details
are bundled/imported from Bruin exports, while schedule rows may be refreshed
from MotherDuck during deployment.

## Bruin Pipeline

Bruin is used as the data orchestrator for ingestion, transformation, metadata,
validation, and export. The project has two Bruin pipelines:

- `pipelines/worldcup_2026` is the live-score pipeline. It is scheduled every
  2 minutes and builds `marts.app_matches` plus `marts.app_live_manifest`.
- `pipelines/worldcup_2026_enrichment` is the enrichment pipeline. It is
  scheduled hourly and builds teams, match details, betting marts, and
  `marts.app_data_manifest`.

Both pipelines target the `fifa` database through the `motherduck-fifa`
connection defined in `.bruin.yml` from `.bruin.yml.example`.

### Bruin Features Used

- **SQL assets**: DuckDB SQL assets ingest local/remote JSON, call public HTTP
  endpoints through DuckDB `read_text`, and build app-facing marts.
- **Python assets**: Python Bruin assets fetch ESPN summaries and Kalshi market
  payloads, then materialize structured raw tables.
- **ingestr YAML assets**: Retained ESPN source assets use Bruin's embedded
  ingestr runner for scoreboard, teams, competitors, standings, and news.
- **Asset metadata**: Assets declare `name`, `description`, `connection`,
  `columns`, primary keys, and data checks such as `not_null`.
- **Materializations**: Assets use table materializations with
  `create+replace` or `truncate+insert`, depending on whether the table should
  be rebuilt or replaced from a stable input.
- **Dependencies and selectors**: Marts declare upstream dependencies. NPM
  scripts use Bruin selectors such as `+marts.app_live_manifest` and
  `+marts.app_data_manifest` so Bruin runs each manifest and everything needed
  upstream.
- **Cross-pipeline URIs**: The enrichment pipeline references live-pipeline
  tables by MotherDuck URI, for example `motherduck://fifa/marts/app_matches`.
- **Pipeline schedules and concurrency**: `pipeline.yml` files define cron
  schedules, `catchup: false`, `concurrency: 1`, `max_active_steps: 1`, and
  retries.
- **Validation**: `npm run pipeline:validate` runs `bruin validate pipelines`.
- **Warehouse export through Bruin CLI**: `npm run pipeline:export` uses
  `bruin query --connection motherduck-fifa --description ...` to read
  MotherDuck marts and serialize app-ready JSON into `src/data/bruin/` and
  `public/data/bruin/`.

### Pipeline Commands

```bash
cp .bruin.yml.example .bruin.yml
export MOTHERDUCK_TOKEN=...

npm run pipeline:validate
npm run pipeline:run
npm run pipeline:export
```

`npm run pipeline:refresh` runs `pipeline:run` and then `pipeline:export`.

`pipeline:run` executes:

- `pipeline:run:live`: `bruin run pipelines/worldcup_2026 --selector
  +marts.app_live_manifest --workers 1`
- `pipeline:run:enrichment`: `bruin run pipelines/worldcup_2026_enrichment
  --selector +marts.app_data_manifest --workers 1`

Set `WORLDCUP_DUCKDB_PATH` only when exporting from a local DuckDB file instead
of MotherDuck. Otherwise exports use Bruin's `motherduck-fifa` connection.

## Data Sources

Bruin lands and transforms these inputs:

- Curated reference schedule JSON for the full 104-match tournament.
- Curated team metadata JSON for team profiles and flags.
- ESPN scoreboard data for event IDs, kickoff alignment, score state, clocks,
  half-time/full-time scores, venue, and goal details.
- ESPN match-summary data for box scores, facts, rosters, and starting XIs.
- Kalshi public World Cup winner and correct-score markets.
- Polymarket public World Cup search results for team futures, group winners,
  knockout qualification, Golden Boot, and continent markets.

The app's scoring, standings, third-place race, bracket labels, Golden Boot,
and comparison stats are computed in React from these generated JSON files.

## MotherDuck

MotherDuck is the shared warehouse for Bruin output and live reads:

- Database: `fifa`.
- Bruin connection name: `motherduck-fifa`.
- Required token: `MOTHERDUCK_TOKEN`.
- Raw schema examples: `raw.reference_schedule_json`,
  `raw.espn_scoreboard_window`, `raw.espn_match_summary`,
  `raw.kalshi_worldcup_events`, `raw.kalshi_worldcup_correct_score`, and
  `raw.polymarket_search_worldcup`.
- Mart schema examples: `marts.app_matches`, `marts.app_teams`,
  `marts.app_team_betting`, `marts.app_match_betting`,
  `marts.app_match_market_summary`, `marts.app_market_insights`,
  `marts.app_live_manifest`, and `marts.app_data_manifest`.

Local exports read MotherDuck through `bruin query`. The Netlify function reads
MotherDuck directly with `@duckdb/node-api` by opening
`md:fifa?motherduck_token=...`. The function's `MOTHERDUCK_DATABASE` variable
can override the database name, but defaults to `fifa`.

MotherDuck is never contacted from browser JavaScript. The browser receives
either static JSON files or a sanitized JSON response from the Netlify function.

## Netlify

`netlify.toml` configures the deployment:

- Build command: `npm run build`.
- Publish directory: `dist`.
- Node version: `20.19.0`.
- Netlify Functions bundled with esbuild.
- `@duckdb/node-api` and `@duckdb/node-bindings` marked as external function
  modules so the live MotherDuck reader can run.
- `/data/bruin/*` headers set `Cache-Control: no-cache, no-store,
  must-revalidate` so refreshed public JSON can be picked up quickly.
- `/assets/*` headers use long immutable caching for Vite-built assets.
- `/*` redirects to `/index.html` with status `200` so the SPA works on
  Netlify.

The main function is `netlify/functions/bruin-schedule.mjs`:

1. Optionally calls `ensureBruinRun()` from `bruin-cloud.mjs`.
2. `ensureBruinRun()` uses `BRUIN_CLOUD_API_KEY` and Bruin Cloud API settings
   to check recent pipeline runs and trigger `worldcup_2026` when the cooldown
   allows.
3. Reads `marts.app_matches`, `marts.app_live_manifest`, and
   `raw.espn_scoreboard_window` from MotherDuck.
4. Converts rows into the same schedule shape used by the bundled JSON.
5. Caches the function payload in memory for 15 seconds and returns `no-store`
   headers to the browser.

Netlify environment variables:

- `MOTHERDUCK_TOKEN` is required for the live schedule function.
- `MOTHERDUCK_DATABASE` optionally overrides `fifa`.
- `DUCKDB_HOME` and `DUCKDB_TEMP_DIRECTORY` optionally control function
  filesystem locations; defaults use `/tmp`.
- `BRUIN_CLOUD_API_KEY` enables on-demand Bruin Cloud triggering.
- `BRUIN_CLOUD_PROJECT_ID`, `BRUIN_CLOUD_PIPELINE`, and
  `BRUIN_CLOUD_BASE_URL` override Bruin Cloud defaults.
- `BRUIN_TRIGGER_COOLDOWN_MS`, `BRUIN_TRIGGER_QUEUE_DEPTH`, and
  `BRUIN_TRIGGER_QUEUE_COOLDOWN_MS` tune function-side trigger throttling.
- `VITE_POSTHOG_KEY` and `VITE_POSTHOG_HOST` control optional browser
  analytics.

If `MOTHERDUCK_TOKEN` or the function is unavailable, the app still works from
the committed Bruin exports in `public/data/bruin/` and the bundled imports in
`src/data/bruin/`.

## Current Tooling Note

Bruin CLI recognizes and executes the ESPN ingestr assets through its embedded
ingestr runner. The standalone `ingestr` binary may reject ESPN or soccer URI
schemes, so run those assets through `bruin run` instead of invoking ingestr
directly.
